BUG

  • 20190801 WebSocketClient Connect 出现 OnOpen 失败后，原逻辑是将继续发送Login 和 Subscribe 信息，这将导致程序异常，使得后续的Client 均无法发送Login和Subscribe信息。

    修复为：1. 添加try catch 保证程序继续进行；2. 在连接OnError 回调函数中移除客户端

  • 20190802，在进行websocketclient 连接建立的时候，会发现websocketclient出现尚未建立连接，会进行登入和订阅消息的发送，导致异常出现

    修复为：采用同步策略，即所有连接均已经建立后再发送其他请求。采用CountLatchDown类，在OnConnected和OnError中进行倒计数，避免其余所有websocket client一直被阻塞

  • 把订阅的命令放在，登入成功后的回调函数处理中；遵循先登入后订阅的原则；【已完成】应该把登入的命令放在connect连接的回调，onOpen函数中，所有流程都同步的话，是不需要进行CountLatchDown同步的
  • 系统初始运行时，数据结构的建立是发生在instrumentInfo第一次出现的时候，比如各种Map数据，但似乎会出现异常情况即：同时多线程InstrumentInfo回调时，会出现NullpointerExceptio； 考虑解决方案是：将容器的初始化放在Subscribe 或者 Login 的回调函数中
  • 新增Socket 中的KeepAlive选项，放在OnOpen中实现 新增在OnClose回调函数中尝试重连接 新增 使用MaxDelay 表示服务器的状态：-1 表示初始化，-2表示已连接，-3表示断开，-4表示错误
